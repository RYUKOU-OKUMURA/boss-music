# Google Drive 連携バグレポート

作成日: 2026-03-23  
対象: boss-music 本番環境の Google Drive 連携と楽曲アップロード

## 1. 概要

本件では、Google Drive 連携および MP3 アップロードまわりで複数の不具合が連鎖していた。

最初の表面症状は以下だった。

- Google Drive 連携時に `ENOENT: no such file or directory, mkdir '/var/task/data'`
- 本番で MP3 をアップロードできない
- ブラウザ側で `Network error`
- Google 認証時に `redirect_uri_mismatch`
- アップロードのたびに毎回 Google の許可画面が出る

最終的に、Vercel 本番での構成を維持したまま、

- サーバー側 OAuth 永続化の不備を解消
- ブラウザから Google Drive へ直接アップロードする構成へ変更
- 同一 Google アカウント前提で再認証負荷を下げる
- 画像アップロードを任意化

まで対応した。

## 2. どうやってバグを見つけたか

### 2-1. 初期エラーの切り分け

最初に、コードベース全体から以下を検索した。

- `/var/task/data`
- Google Drive 連携処理
- トークン保存処理
- Vercel / Redis 関連の環境変数

その結果、Google OAuth コールバック後に `refresh_token` を保存する処理が、Redis 未設定時に `data/drive-tokens.enc` へ保存しようとしていることを確認した。

該当箇所:

- `server/routes/auth.ts`
- `server/services/tokenStore.ts`
- `server/services/redisStore.ts`

### 2-2. 本番アップロード不能の切り分け

次に、管理画面からのアップロード経路を確認した。

確認した内容:

- `src/components/Admin.tsx`
- `server/routes/admin.ts`
- `server/utils/upload.ts`

その結果、当初の実装は

1. ブラウザがファイルを `/api/admin/upload` に送る
2. Vercel Function が `multer.memoryStorage()` で受ける
3. サーバーから Google Drive に転送する

という構成だった。

この構成は、Vercel のサーバーレス制約と相性が悪い。

さらに、その次の修正案として一度導入した

- サーバーが Google Drive の resumable upload session を発行
- ブラウザがその session URL に直接 `PUT`

という中間構成では、ブラウザ側で `Network error` が発生した。

この時点で、

- Google 認証は通っている
- しかしブラウザから Google Drive への直送が成立していない

という点を切り分けた。

### 2-3. OAuth 設定不一致の切り分け

その後、Google の認証画面で `redirect_uri_mismatch` が出た。

ここでは、以下を確認した。

- Vercel の環境変数 `OAUTH_REDIRECT_URI`
- Google Cloud Console の OAuth クライアント設定
- Authorized JavaScript origins
- Authorized redirect URIs

その結果、

- 本番 URL に対する JavaScript origin
- `/api/auth/google/callback`

の両方が Google 側設定と一致していないことが原因と分かった。

### 2-4. 毎回許可が出る問題の切り分け

アップロード成功後に、毎回 Google の許可が出る問題を確認した。

管理画面コードを確認したところ、Google Identity Services の token client を毎回

- `prompt: 'consent select_account'`

で起動していたため、実装上こちらが毎回同意を強制していた。

## 3. 原因

今回の原因は単一ではなく、主に4つあった。

### 原因1. Vercel 本番でローカルファイル保存にフォールバックしていた

Google OAuth コールバック後の `refresh_token` 保存が、Redis 未設定時にローカルファイル保存へ落ちていた。

Vercel では実行パスが `/var/task` になるため、`/var/task/data` を作ろうとして失敗していた。

実質的な原因:

- 本番で Redis 必須なのに、コードがローカル保存へフォールバックしていた
- しかも Redis 判定に使う環境変数名と、実際に Vercel に入っていた変数名が一致していなかった

### 原因2. 本番アップロード構成が Vercel に適していなかった

最初の構成では、MP3 を一度 Vercel Function に送ってから Drive に転送していた。

この構成は、

- リクエストサイズ
- 実行時間
- メモリ使用

の面で不安定になりやすく、本番で「確実にアップロードする」要件に向いていなかった。

### 原因3. Google OAuth のクライアント設定が本番 URL と一致していなかった

ブラウザ側 OAuth を導入した後、Google Cloud Console の OAuth クライアントで

- Authorized JavaScript origins
- Authorized redirect URIs

が本番 URL と一致していなかったため、`redirect_uri_mismatch` が発生した。

### 原因4. Google 認証を毎回強制する設定にしていた

Google Identity Services の token client に `consent select_account` を毎回指定していたため、同じアカウントでも毎回許可画面が出ていた。

## 4. 原因を解消するために考えたこと

### 4-1. Vercel を残す前提を維持する

ユーザー要望として、

- Vercel から API を切り分けたくない
- 同じ Google アカウントで運用する

という制約があった。

そのため、サーバーを Railway / Render / Cloud Run に逃がす案は採らず、

- Vercel は継続
- アップロードはブラウザ直送

に方針を固定した。

### 4-2. サーバーとブラウザの責務を分ける

安定化のために、役割を以下のように分離した。

- サーバー
  - Drive 連携状態確認
  - 連携済み Google アカウント情報の取得
  - アップロード完了後の fileId 検証
  - catalog 更新

- ブラウザ
  - Google Identity Services によるクライアント OAuth
  - Google Drive への直接アップロード

### 4-3. 同じ Google アカウント前提を活かす

アップロード用アカウントを毎回変えない前提だったため、

- サーバー側の連携済みアカウント
- ブラウザで選択した Google アカウント

が一致しているか確認する仕様にした。

これにより、

- 別アカウントに誤アップロードする事故を防ぐ
- `select_account` の強制を減らせる

という効果を狙った。

## 5. 実際に施した修正

### 修正1. 本番で Redis 未設定なら明示エラーに変更

追加:

- `server/services/runtimeEnv.ts`

変更:

- `server/services/tokenStore.ts`
- `server/services/catalog.ts`
- `server/routes/admin.ts`
- `server/routes/tracks.ts`
- `server/routes/auth.ts`

内容:

- Vercel 本番で `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` が無い場合、ローカル保存に落ちず設定エラーで止めるようにした
- `drive-status` で `storage`, `configOk`, `reason` を返すようにした

### 修正2. アップロード経路をブラウザ直送へ変更

変更:

- `src/components/Admin.tsx`
- `server/routes/admin.ts`
- `server/services/driveClient.ts`
- `server/services/driveUploads.ts`

内容:

- 旧 `/api/admin/upload` の主経路利用を停止
- 管理画面で Google Identity Services を使って Drive 用アクセストークンを取得
- ブラウザから Google Drive API に直接アップロード
- アップロード完了後に `/api/admin/upload/complete` で fileId を検証して catalog を更新

### 修正3. Google OAuth 設定不一致の再発を防ぐための文言改善

変更:

- `src/components/Admin.tsx`
- `README.md`

内容:

- `redirect_uri_mismatch`
- `origin_mismatch`
- popup closed

などをまとめて、Google Cloud Console のどこを直すべきか分かるエラーメッセージに変更した。

### 修正4. 毎回の許可要求を緩和

変更:

- `src/components/Admin.tsx`

内容:

- `prompt: 'consent select_account'` を毎回強制しないよう変更
- 同一アカウント前提で `login_hint` を利用
- 取得済みアクセストークンを `sessionStorage` にキャッシュ
- 必要な場合だけ再同意にフォールバック

結果:

- 初回のみ許可
- 同じブラウザセッション中は再同意なしで再アップロード可能

### 修正5. 画像アップロードを任意化

変更:

- `src/components/Admin.tsx`
- `server/routes/admin.ts`
- `server/services/catalog.ts`
- `server/utils/trackPublic.ts`
- `src/lib/mediaUrls.ts`
- `src/context/AudioContext.tsx`
- `src/components/TrackCard.tsx`
- `src/components/VinylDisc.tsx`

内容:

- 画像入力を必須から任意に変更
- `driveCoverFileId` を optional に変更
- 画像がない曲でも公開一覧・詳細画面が崩れないよう、プレースホルダ表示を追加

## 6. 結果

最終的に以下を確認した。

- Google Drive 連携成功
- Vercel 本番で MP3 アップロード成功
- 曲の再生成功
- 同じ Google アカウント前提で再アップロード時の許可要求を削減
- 画像なしでも楽曲登録可能

## 7. 今回の学び

### 7-1. サーバーレス環境では「ローカル保存フォールバック」が危険

開発環境では便利でも、本番サーバーレスでは誤動作の原因になりやすい。  
本番必須要件は、コード上も「必須」として止めるべき。

### 7-2. Google OAuth は「サーバー用」と「ブラウザ用」で見る設定が違う

今回のように両方を使う場合、

- Authorized JavaScript origins
- Authorized redirect URIs

のどちらが効いている問題かを切り分ける必要がある。

### 7-3. UX 上の不満も実装起因であることが多い

「毎回許可を求められる」は Google の仕様に見えやすいが、実際は `prompt` 指定とトークン再利用戦略の問題だった。

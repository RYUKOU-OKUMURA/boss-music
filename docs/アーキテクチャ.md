プロジェクト: Suno 楽曲ポートフォリオサイト（boss-music）
最終更新: 2026-04-24

## 2. データの正本

| 種別 | 正本 | 補足 |
|------|------|------|
| 曲メタデータ | Neon Postgres `tracks` テーブル | タイトル、アーティスト、説明、プレイリスト、タグ、Blob URL / pathname など |
| 音声ファイル | Vercel Blob Public Store | MP3。公開再生は Blob の HTTPS URL を直接使う |
| ジャケット画像 | Vercel Blob Public Store | JPG / PNG / WEBP |
| 管理者認証 | `ADMIN_SECRET` または `SESSION_SECRET` 署名 Cookie | 管理 API のみ保護 |

`tracks` テーブルは API 起動時の `ensureTracksSchema()` で自動作成される。手動作成用 SQL は `migrations/001_create_tracks.sql`。

## 3. デプロイ形態

| 形態 | エントリ | 静的ファイル | API |
|------|----------|--------------|-----|
| ローカル開発 | `npm run dev:all` | Vite `:3000` | Express `:8787` |
| 単一プロセス本番 | `npm run build` → `NODE_ENV=production npm start` | Express が `dist/` を配信 | 同一 Express |
| Vercel | `npm run build` | `dist/` を Vercel が配信 | `/api/*` を `api/index.mjs` へ rewrite |

Vercel では `api/_handler.ts` を esbuild で `api/index.mjs` にバンドルし、`createApiApp()` を default export する。`vercel.json` は `/api/(.*)` を `/api` へ寄せ、その他のパスを SPA の `index.html` へ rewrite する。

## 5. API エンドポイント

| メソッド | パス | 認証 | 役割 |
|----------|------|------|------|
| GET | `/api/tracks` | 不要 | Neon の `tracks` を公開用 `Track` 配列に変換して返す |
| GET | `/api/health` | 不要 | `{ ok: true }` |
| GET | `/api/admin/storage-status` | 不要 | `DATABASE_URL` / `BLOB_READ_WRITE_TOKEN` と疎通を確認 |
| POST | `/api/admin/session` | `X-Admin-Secret` | `SESSION_SECRET` がある場合に管理 Cookie を発行 |
| POST | `/api/admin/blob-upload` | 管理者 | Vercel Blob client upload 用 token / response を処理 |
| POST | `/api/admin/upload/complete` | 管理者 | Blob metadata を検証し、Neon に曲を登録 |
| POST | `/api/admin/upload` | 管理者 | 旧 multipart 経路。現在は `410 Gone` |
| POST | `/api/admin/tracks/:id/playlist` | 管理者 | プレイリストを更新 |
| POST | `/api/admin/tracks/:id/cover` | 管理者 | ジャケット画像を差し替え |
| DELETE | `/api/admin/tracks/:id/cover` | 管理者 | ジャケット画像を削除 |
| DELETE | `/api/admin/tracks/:id` | 管理者 | 曲レコードと Blob ファイルを削除 |

## 6. 管理アップロードの流れ

1. 管理者が `/admin` を開く。
2. 管理画面が `/api/admin/storage-status` で Blob / DB 設定を確認する。
3. MP3 と任意のジャケット画像を選択する。
4. ブラウザが `/api/admin/blob-upload` を介して Vercel Blob client upload を実行する。
5. アップロード完了後、ブラウザが `/api/admin/upload/complete` に Blob metadata と曲情報を送る。
6. API が Blob の存在・pathname・size・contentType を検証する。
7. API が Neon `tracks` に曲を登録する。
8. フロントは `boss-music-catalog-changed` イベントで一覧を再取得する。

旧 `/api/admin/upload` の multipart 経路は廃止済み。大きい MP3 を Serverless Function に丸ごと送らず、ブラウザから Blob へ直接送る。

## 7. フロントエンド

| URL | コンポーネント | 役割 |
|-----|----------------|------|
| `/` | `Gallery` | 曲一覧、プレイリストフォルダ、検索、再生開始 |
| `/track/:id` | `TrackPage` | 曲詳細、再生 UI、レイアウト切替 |
| `/admin` | `Admin` | アップロード、カバー差し替え、削除、プレイリスト変更 |

`AudioProvider` はアプリ全体で 1 つの `<audio>` を管理する。`AudioMainContext` と `AudioTimeContext` を分け、高頻度更新の `currentTime` が不要なコンポーネントの再レンダーを抑えている。

---

## 8. メディア URL

現行の公開トラックは `audioUrl` と `coverImage` に Vercel Blob の HTTPS URL を持つ。`resolveAudioPlaybackUrl()` は古い相対 URL なども扱えるが、現在の主経路は Blob URL の直接再生。

`audioCache.ts` は Safari / iOS などで直接ストリーム再生が不安定な場合に備え、音声を Blob として取得して `blob:` URL で再生するための補助。

## 9. 環境変数

必須:

- `DATABASE_URL`
- `BLOB_READ_WRITE_TOKEN`
- `ADMIN_SECRET`

推奨:

- `SESSION_SECRET`

開発時のみ任意:

- `VITE_ADMIN_SECRET`
- `VITE_API_BASE_URL`
- `API_PORT`

本番では `VITE_ADMIN_SECRET` を埋め込まない。フロントに入る `VITE_*` は公開値として扱う。
